# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-06-12
**Phases:** 1–3
**Total Plans:** 7

## Overview

Initial portfolio-ready release of the German Clinical NLP Pipeline. A complete information extraction pipeline that takes German clinical text and returns structured JSON with dates, diagnoses, medications, and length-of-stay indicators — deployed via Docker with one-command setup. All 29 v1 requirements delivered across 3 phases in a single day.

**Goal:** Complete extraction pipeline that extracts all entity types from German clinical text with validated JSON output
**Depends on:** Phase 1
**Plans:** 3 plans
**Requirements covered:** ENTITY-01 through ENTITY-06, VAL-01, VAL-02, VAL-03, API-01, API-04, EXT-01, EXT-02

Plans:
- [x] 02-01-PLAN.md — Temporal entity extraction with plugin pattern foundation (completed 2026-06-12)
- [x] 02-02-PLAN.md — Clinical entity extraction with parallel execution (completed 2026-06-12)
- [x] 02-03-PLAN.md — Validation logic and confidence filtering (completed 2026-06-12)

**Details:**

Success criteria:
1. ✅ User can POST German clinical text to /extract and receive JSON with dates, diagnoses, medications, and length-of-stay indicators
2. ✅ Each extracted entity includes confidence score and source text span (evidence grounding)
3. ✅ Pydantic validation rejects impossible values (future dates, invalid formats, hallucinated entities without source spans)
4. ✅ API returns meaningful error messages for invalid requests
5. ✅ Developer can add a new entity type by creating a single extractor plugin without modifying core pipeline code

**Notable:** Plugin registry pattern with @register_extractor decorator. asyncio.gather parallel extraction via asyncio.to_thread (llama-cpp-python is synchronous). Source span validation detects LLM hallucination. Configurable CONFIDENCE_THRESHOLD env var with fail-fast validation.

## Milestone Summary

**Key Decisions:**

| Decision | Rationale |
|----------|-----------|
| Synthetic German clinical samples | HuggingFace bigbio/ggponc2 deprecated loading scripts (datasets 3.0+) |
| Plugin registry with @register_extractor | Clean extensibility, no core code changes to add entity types |
| asyncio.to_thread for parallel extraction | llama-cpp-python is synchronous; threads enable async gather |
| Confidence filtering at endpoint level (D-20) | Respect configurable CONFIDENCE_THRESHOLD env var; extractors are pure |
| Zero-based character offsets [start, end) (D-13) | Python slicing convention, simplifies span validation |
| Single-stage Dockerfile from python:3.11-slim (D-01) | Simpler than multi-stage for portfolio demo, build-essential fallback for wheel |
| stdlib urllib healthcheck (D-02) | No curl in slim image, zero overhead |

**Issues Resolved:**

- Fixed .gitignore `models/` pattern blocking `src/models/` package (changed to `/models/`)
- Removed hardcoded confidence=0.5 threshold from extractors (D-20 consolidation)
- Added loguru as declared runtime dep (was imported but missing from pyproject.toml)

**Issues Deferred:**

- Real GGUF model integration tests (requires 4.5-40GB download, deferred to developer setup)
- GGPONC corpus loading (HuggingFace API incompatibility, synthetic samples used)
- Comprehensive automated test coverage (v2 QUAL-04)

**Technical Debt Incurred:**

- Example JSON files are hand-crafted, not live model output (noted in _generated_by field)
- Windows build environment lacks C++ toolchain for llama-cpp-python (tests use mocked model)
